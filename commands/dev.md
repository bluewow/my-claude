# Dev (Fullstack 구현 에이전트)

**brief.md를 읽고, 범위 파악 → 계획 → 구현 → 에러 체크까지 진행합니다.**
**백그라운드 Agent로 실행되므로, 메인 대화는 계속 사용할 수 있습니다.**

---

## 실행 방법

1. `output/tasks/$ARGUMENTS/brief.md` 파일이 존재하는지 확인
2. 존재하지 않으면 "brief.md가 없습니다. `/experiment:pm`으로 먼저 기획하세요." 안내 후 중단
3. 존재하면 **백그라운드 Agent를 생성**하여 전체 파이프라인 실행

### Agent 생성

```
Agent(
  description: "dev: $ARGUMENTS",
  prompt: 아래 프롬프트 (brief 경로를 포함),
  run_in_background: true,
  mode: "auto"
)
```

### Agent 프롬프트

아래 내용을 Agent의 prompt로 전달한다. `{TASK_ID}`는 `$ARGUMENTS`로, `{BRIEF_PATH}`와 `{TASK_FOLDER}`는 실제 절대 경로로 치환한다.

**수정 사항이 있는 경우**: PM에서 `__DEV_MODIFICATIONS__`로 전달된 수정 피드백이 있으면, `{MODIFICATIONS}` 섹션을 Agent 프롬프트에 추가한다. 없으면 해당 섹션을 생략한다.

## Step 1: 범위 파악

1. 루트 CLAUDE.md를 읽고 프로젝트 기술 스택, 구조, 패턴 파악
2. `{BRIEF_PATH}`를 읽고 기획 내용 파악
3. CLAUDE.md의 Project Structure를 기반으로 프로젝트 루트 디렉토리 확인
4. 프로젝트 전체를 스캔하여 관련 영역 파악:
   - 프론트엔드: 컴포넌트, 화면, 라우팅, 상태 관리, 스타일 토큰
   - 백엔드: API, 모델, 서비스, 데이터베이스
   - 공통: 유틸, 타입 정의, 설정 파일
5. brief.md의 "기존 자산" 섹션과 실제 코드를 대조하여 재사용 대상 확정
6. 프론트-백엔드 간 연결 지점 파악 (API 호출, 데이터 모델 공유 등)

## Step 2: 계획

범위 파악 결과를 바탕으로 구현 순서를 결정합니다:

1. **변경 파일 목록** — 수정/생성/삭제할 파일 정리
2. **구현 순서 결정** — 의존성을 고려한 작업 순서
   - 데이터 모델/타입 정의 먼저
   - 백엔드 (API, 서비스) → 프론트엔드 (화면, 컴포넌트) 순서
   - 또는 프로젝트 구조에 따라 적절히 판단
3. **위험 요소** — 기존 코드에 영향 줄 수 있는 부분 식별

계획은 파일로 저장하지 않고, 이 컨텍스트 안에서 유지합니다.

## Step 3: 구현

계획 순서대로 구현합니다:

### 규칙
- CLAUDE.md에 정의된 패턴/컨벤션을 따를 것
- 기존 컴포넌트/서비스가 있으면 반드시 재사용 (중복 생성 금지)
- 새 코드는 기존 코드와 동일한 구조/패턴으로 생성
- 기존 스타일 토큰 사용, 새 토큰 추가는 최소화
- 프론트-백엔드 간 데이터 타입/인터페이스 일관성 유지
- 중간에 멈추지 말고 완료까지
- 문제 발생 시에만 멈추고 보고

## Step 4: 에러 체크

구현 완료 후 반드시 검증합니다:

1. **정적 분석** — CLAUDE.md의 Dev Commands에 analyze 명령이 있으면 실행
2. **빌드 확인** — 프로젝트 빌드가 성공하는지 확인
3. **테스트** — 기존 테스트가 있으면 실행하여 깨진 테스트 확인
4. **에러 수정** — 위 단계에서 발견된 에러를 수정
5. **재검증** — 에러 수정 후 다시 빌드/테스트 통과 확인

에러가 3회 이상 반복되면 수정을 중단하고 complete.md에 미해결 이슈로 기록합니다.

## Step 5: 완료 보고

`{TASK_FOLDER}/complete.md`에 결과를 저장합니다:

```markdown
# 구현 완료: [기능명]
> 태스크: {TASK_ID}
> 완료일: [날짜]

## 변경 사항
- 생성: [파일 목록]
- 수정: [파일 목록]
- 삭제: [파일 목록]

## 구현 요약
[주요 변경 사항 요약 3~5줄]

## 에러 체크 결과
- 정적 분석: [통과/실패]
- 빌드: [통과/실패]
- 테스트: [통과/실패/해당없음]

## 미해결 이슈 (있는 경우)
- [이슈 설명]

## 확인 필요
[수동 테스트 필요 사항 등]
```

## Step 6: history.json 업데이트 ← 반드시 실행, 건너뛰기 금지

complete.md 저장 후 즉시 수행합니다. 이 단계를 빠뜨리면 대시보드에 완료 상태가 반영되지 않습니다.

실행 순서:
1. `output/history.json` Read
2. `{TASK_ID}`에 해당하는 태스크의 `logs` 배열에 아래 항목 추가:
   ```json
   { "phase": "개발", "status": "완료", "date": "[날짜]", "note": "구현 + 에러체크 완료" }
   ```
3. 해당 태스크의 `artifacts` 배열에 `"complete.md"` 추가 (이미 없는 경우만)
4. Write로 저장
5. Read로 재확인 — logs와 artifacts가 실제로 추가됐는지 검증
```

## 실행 후 메인 대화에 출력할 메시지

Agent 생성 후 메인 대화에 다음을 출력한다:

```
백그라운드에서 개발을 시작했습니다.
- 태스크: $ARGUMENTS
- 입력: output/tasks/$ARGUMENTS/brief.md
- 파이프라인: 범위 파악 → 계획 → 구현 → 에러 체크
- 산출물: complete.md (완료 시)

이 대화에서 다른 작업을 계속하실 수 있습니다.
완료되면 알림이 옵니다.
```
