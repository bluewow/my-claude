# Do (직접 위임 에이전트)

**기획이 필요 없는 작업을 직접 위임합니다.**
**버그 수정, 간단한 변경, 텍스트 수정 등 "뭘 해야 하는지 이미 아는" 작업에 사용합니다.**
**백그라운드 Agent로 실행되므로, 메인 대화는 계속 사용할 수 있습니다.**

---

## 사용 예시

```
/experiment:do 대시보드 카드 클릭 시 에러 발생
/experiment:do 필터 종류를 날짜순/이름순/상태순으로 변경
/experiment:do 사이드바 메뉴 텍스트 "프로젝트" → "워크스페이스"로 변경
/experiment:do [이미지 붙여넣기] 이 화면에서 레이아웃 깨짐
/experiment:do C:/screenshots/bug.png 참고해서 수정
```

---

## 실행 방법

1. **사전 트리아지** (foreground) — 빠른 스캔 + 필요 시 사용자 확인
2. `output/history.json`을 읽어 마지막 ID 확인 → 새 태스크 ID 채번
3. `output/tasks/[ID]-[slug]/` 폴더 생성
4. `output/history.json`에 태스크 엔트리 추가
5. **백그라운드 Agent를 생성**하여 전체 파이프라인 실행

### Step 0: 사전 트리아지 (foreground — 백그라운드 진입 전)

Agent 생성 **전에** 메인 대화에서 직접 수행한다. 빠르게 끝내는 것이 목적이므로 30초 이내로 완료한다.

#### 0-1. 빠른 스캔

1. 루트 CLAUDE.md를 읽어 프로젝트 구조 파악
2. `$ARGUMENTS`에서 키워드를 추출하여 Grep/Glob으로 관련 코드 탐색 (최대 3회)
3. 이미지가 포함되어 있으면 Read로 확인

#### 0-2. 복잡도 판정

스캔 결과를 기반으로 작업 복잡도를 판정한다:

| 판정 | 조건 | 행동 |
|:-----|:-----|:-----|
| **단순** | 수정 대상이 명확하고, 수정 방향이 1가지 | → 바로 백그라운드 실행 |
| **정보 부족** | 원인 파악에 추가 정보가 필요 (에러 메시지, 재현 조건, 대상 특정 불가) | → 0-3으로 |
| **선택 필요** | 수정 방향이 2가지 이상 존재 | → 0-4로 |

#### 0-3. 추가 정보 수집 (정보 부족일 때만)

AskUserQuestion으로 부족한 정보를 질문한다. 한 번에 묶어서 질문하되, 최대 3개까지만.

질문 예시:
- "어떤 화면/페이지에서 발생하나요?"
- "에러 메시지나 콘솔 로그가 있나요?"
- "특정 조건에서만 발생하나요? (예: 로그인 후, 특정 데이터)"

답변을 받은 후 `{TASK_DESCRIPTION}`에 수집된 정보를 추가하여 Agent에 전달한다.

#### 0-4. 수정 방향 선택 (선택지가 있을 때만)

수정 방향이 여러 개일 때, 각 방향의 요약과 영향 범위를 제시한다.

AskUserQuestion:
```
question: "수정 방향 선택"
options:
  - label: "[방향 A 요약]"
    description: "[영향 범위 + 장단점 한줄]"
  - label: "[방향 B 요약]"
    description: "[영향 범위 + 장단점 한줄]"
  - label: "알아서 판단"
    description: "에이전트가 최적의 방향을 선택합니다"
```

선택된 방향을 `{TASK_DESCRIPTION}`에 추가하여 Agent에 전달한다.
"알아서 판단"을 선택하면 기존처럼 Agent가 자체 결정한다.

### ID 채번

- ID 형식: `[3자리숫자]-[slug]` (예: `008-fix-card-click`, `009-change-filter`)
- slug는 `$ARGUMENTS`에서 핵심 키워드를 추출하여 생성

### history.json 엔트리 추가

```json
{
  "id": "[ID]-[slug]",
  "slug": "[slug]",
  "name": "[작업 요약]",
  "path": "output/tasks/[ID]-[slug]",
  "artifacts": [],
  "logs": [
    {
      "phase": "작업",
      "status": "진행중",
      "date": "[오늘 날짜 YYYY-MM-DD]",
      "note": "[작업 설명 요약]"
    }
  ]
}
```

### Agent 생성

```
Agent(
  description: "do: [작업 요약 5단어]",
  prompt: 아래 프롬프트,
  run_in_background: true,
  mode: "auto"
)
```

### Agent 프롬프트

아래 내용을 Agent의 prompt로 전달한다.
- `{TASK_DESCRIPTION}`: `$ARGUMENTS` + 트리아지에서 수집된 추가 정보 (있는 경우)
- `{TRIAGE_CONTEXT}`: 트리아지 결과 요약 (관련 파일, 선택된 수정 방향 등). 트리아지에서 특별한 컨텍스트가 없으면 빈 문자열.
- `{TASK_ID}`, `{TASK_FOLDER}`: 실제 값으로 치환
- `{TASK_SLUG}`: ID에서 숫자 제외한 slug 부분
- `{TASK_NAME}`: 작업 요약 (한 줄)
- `{TODAY}`: 오늘 날짜 YYYY-MM-DD
- `{TASK_DESCRIPTION_SHORT}`: 작업 설명 30자 이내 요약

---

```
당신은 풀스택 개발자입니다. 요청받은 작업을 분석하고 수행합니다.

## 작업 내용

{TASK_DESCRIPTION}

## 트리아지 컨텍스트

{TRIAGE_CONTEXT}

## Step 0: history.json 초기 엔트리 보장 ← 반드시 먼저 실행

백그라운드 에이전트 시작 직후 즉시 수행합니다.

1. `output/history.json` Read
2. `{TASK_ID}` 엔트리가 없으면 배열 끝에 추가:
   ```json
   {
     "id": "{TASK_ID}",
     "slug": "{TASK_SLUG}",
     "name": "{TASK_NAME}",
     "path": "output/tasks/{TASK_ID}",
     "artifacts": [],
     "logs": [
       { "phase": "작업", "status": "진행중", "date": "{TODAY}", "note": "{TASK_DESCRIPTION_SHORT}" }
     ]
   }
   ```
3. Write로 저장
4. Read로 재확인

엔트리가 이미 있으면 건너뜁니다.

## Step 1: 범위 파악

1. 루트 CLAUDE.md를 읽고 프로젝트 기술 스택, 구조, 패턴 파악
2. 이미지가 포함되어 있으면 확인하여 현재 상태 파악
3. 작업 설명에서 키워드를 추출하여 관련 코드 탐색
4. **디자인 레퍼런스 로드** — UI/스타일/레이아웃 관련 작업인 경우 반드시 수행:
   - `.claude/skills/design-inspiration/references/DESIGN.md`를 Read (디자인 시스템 규칙)
   - `.claude/skills/design-inspiration/images/` 폴더의 이미지 파일을 Glob으로 탐색 후 Read (레퍼런스 이미지)
   - 이미지가 없으면 DESIGN.md만으로 진행
   - 수정 시 DESIGN.md의 색상·타이포·엘리베이션·컴포넌트 규칙을 준수할 것
5. 프로젝트를 스캔하여 관련 영역 파악:
   - 프론트엔드: 관련 컴포넌트, 화면, 상태 관리
   - 백엔드: 관련 API, 서비스, 데이터 모델
   - 프론트-백엔드 간 연결 지점
6. 변경 대상과 영향 범위 확인
7. 수정 방안 결정 — 최소 변경으로 작업하는 방법 결정

## Step 2: 구현

### 규칙
- CLAUDE.md에 정의된 패턴/컨벤션을 따를 것
- 최소 변경 원칙 — 요청된 작업에 필요한 코드만 변경
- 기존 코드 구조를 훼손하지 않을 것
- 불필요한 리팩토링 금지
- 기존 컴포넌트/서비스가 있으면 반드시 재사용
- 프론트-백엔드 양쪽에 걸친 작업이라면 양쪽 모두 수정
- 중간에 멈추지 말고 완료까지
- 문제 발생 시에만 멈추고 보고

## Step 3: 에러 체크

구현 완료 후 반드시 검증합니다:

1. **정적 분석** — CLAUDE.md의 Dev Commands에 analyze 명령이 있으면 실행
2. **빌드 확인** — 프로젝트 빌드가 성공하는지 확인
3. **테스트** — 기존 테스트가 있으면 실행하여 깨진 테스트 확인
4. **에러 수정** — 위 단계에서 발견된 에러를 수정
5. **재검증** — 에러 수정 후 다시 빌드/테스트 통과 확인

에러가 3회 이상 반복되면 수정을 중단하고 complete.md에 미해결 이슈로 기록합니다.

## Step 4: 완료 보고

`{TASK_FOLDER}/complete.md`에 결과를 저장합니다:

```markdown
# 작업 완료: [작업 요약]
> 태스크: {TASK_ID}
> 완료일: [날짜]

## 변경 사항
- 생성: [파일 목록]
- 수정: [파일 목록]
- 삭제: [파일 목록]

## 작업 요약
[주요 변경 사항 요약 2~3줄]

## 에러 체크 결과
- 정적 분석: [통과/실패]
- 빌드: [통과/실패]
- 테스트: [통과/실패/해당없음]

## 미해결 이슈 (있는 경우)
- [이슈 설명]

## 확인 방법
[수정 확인을 위한 방법]
```

## Step 5: history.json 업데이트 ← 반드시 실행, 건너뛰기 금지

complete.md 저장 후 즉시 수행합니다. 이 단계를 빠뜨리면 대시보드에 완료 상태가 반영되지 않습니다.

실행 순서:
1. `output/history.json` Read
2. `{TASK_ID}`에 해당하는 태스크의 `logs` 배열에 아래 항목 추가:
   ```json
   { "phase": "작업", "status": "완료", "date": "[날짜]", "note": "구현 + 에러체크 완료" }
   ```
3. 해당 태스크의 `artifacts` 배열에 `"complete.md"` 추가 (이미 없는 경우만)
4. Write로 저장
5. Read로 재확인 — logs와 artifacts가 실제로 추가됐는지 검증
```

## 실행 후 메인 대화에 출력할 메시지

Agent 생성 후 메인 대화에 다음을 출력한다:

```
백그라운드에서 작업을 시작했습니다.
- 태스크: [ID]-[slug]
- 작업: $ARGUMENTS (요약)
- 트리아지: [단순 → 바로 실행 | 추가 정보 수집 완료 | 방향 선택: OOO]
- 파이프라인: 범위 파악 → 구현 → 에러 체크
- 산출물: complete.md (완료 시)

이 대화에서 다른 작업을 계속하실 수 있습니다.
완료되면 알림이 옵니다.
```
